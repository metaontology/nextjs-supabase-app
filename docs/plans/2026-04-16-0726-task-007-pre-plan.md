# Task 007: 데이터베이스 스키마 및 Supabase 초기 설정 — 구현 계획

## Context

Phase 2 (UI/UX 완성)에서 더미 데이터로 모든 페이지 UI가 완성되었으므로, 이제 실제 데이터베이스 스키마를 확정하고 Supabase에 적용한다. 현재 DB에는 `profiles` 테이블만 존재하며 `events`, `event_participants` 테이블 없이 임시 프론트엔드 타입(`lib/types/event.ts`, `lib/types/user.ts`)을 사용 중이다.

이 Task 완료 후 Task 008 (인증), Task 009 (이벤트 CRUD)에서 실제 Supabase 쿼리를 작성할 수 있게 된다.

---

## 핵심 설계 결정

### 필드명: `date` vs `event_date`

- PRD는 `event_date` 사용, 현재 임시 타입·UI 7개 파일은 `event.date` 참조
- **결정: DB 컬럼명 `event_date` 채택** (PRD 준수, 명확한 네이밍)
- 타입 교체 시 `event_date`로 일괄 수정

### `max_participants` 필드

- PRD에 없지만 폼/타입에 이미 구현됨 → **DB에 포함** (INTEGER NULLABLE)

### `status` 컬럼

- CHECK 제약조건으로 `('upcoming', 'ongoing', 'ended')` 강제

### `event_participants.role`

- PRD에 포함 → **DB에 포함** (TEXT DEFAULT 'attendee')

## 구현 단계

### Step 1: profiles 테이블에 role 컬럼 추가

**마이그레이션 파일**: `supabase/migrations/20260416000000_add_role_to_profiles.sql`

```sql
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS role TEXT NOT NULL DEFAULT 'user'
    CHECK (role IN ('user', 'admin'));
```

- 기존 RLS 정책 유지 (SELECT/UPDATE)
- 관리자 전체 조회 정책 추가:
  ```sql
  CREATE POLICY "admins can view all profiles"
    ON public.profiles FOR SELECT
    USING (
      EXISTS (
        SELECT 1 FROM public.profiles p
        WHERE p.id = auth.uid() AND p.role = 'admin'
      )
    );
  ```

### Step 2: events 테이블 생성

**마이그레이션 파일**: `supabase/migrations/20260416000001_create_events_table.sql`

```sql
CREATE TABLE IF NOT EXISTS public.events (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  title            TEXT        NOT NULL,
  description      TEXT,
  event_date       TIMESTAMPTZ NOT NULL,
  location         TEXT        NOT NULL,
  cover_image_url  TEXT,
  invite_code      TEXT        UNIQUE NOT NULL,
  max_participants INTEGER,
  status           TEXT        NOT NULL DEFAULT 'upcoming'
                     CHECK (status IN ('upcoming', 'ongoing', 'ended')),
  created_by       UUID        NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**RLS 정책**:

- `SELECT`: 모든 인증 사용자 (초대 링크 접근 등)
- `INSERT`: 인증 사용자, `created_by = auth.uid()` 강제
- `UPDATE`: 주최자 본인만
- `DELETE`: 주최자 본인만

**인덱스**:

- `events_invite_code_idx ON events (invite_code)` — 초대 코드 조회 최적화
- `events_created_by_idx ON events (created_by)` — 내 이벤트 목록 최적화
- `events_status_idx ON events (status)` — 상태 필터링 최적화

**트리거**: `handle_updated_at()` 재사용 (profiles에서 이미 생성됨)

### Step 3: event_participants 테이블 생성

**마이그레이션 파일**: `supabase/migrations/20260416000002_create_event_participants_table.sql`

```sql
CREATE TABLE IF NOT EXISTS public.event_participants (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id   UUID        NOT NULL REFERENCES public.events(id) ON DELETE CASCADE,
  user_id    UUID        NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  role       TEXT        NOT NULL DEFAULT 'attendee'
               CHECK (role IN ('attendee', 'organizer')),
  joined_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (event_id, user_id)  -- 중복 참여 방지
);
```

**RLS 정책**:

- `SELECT`: 인증 사용자 (이벤트 참여자 목록 조회)
- `INSERT`: 인증 사용자, `user_id = auth.uid()` 강제
- `DELETE`: 본인 참여 기록만

**인덱스**:

- `event_participants_event_id_idx ON event_participants (event_id)` — 이벤트별 참여자 목록
- `event_participants_user_id_idx ON event_participants (user_id)` — 사용자별 참여 이벤트 목록

### Step 4: Storage 버킷 생성

Supabase MCP (`mcp__supabase__execute_sql`)로 생성하거나 대시보드에서 수동 생성:

- 버킷명: `event-covers`
- 공개 버킷 (이미지 URL로 직접 접근)
- 파일 크기 제한: 5MB
- 허용 MIME 타입: `image/jpeg`, `image/png`, `image/webp`

Storage 정책:

- 읽기: 모든 사용자 (public)
- 업로드: 인증 사용자만, 경로 `{user_id}/{filename}` 강제

### Step 5: database.types.ts 업데이트

Supabase CLI로 재생성:

```bash
npx supabase gen types typescript --project-id <project-id> > lib/types/database.types.ts
```

또는 MCP `mcp__supabase__list_tables`로 스키마 확인 후 수동 업데이트.

새로 추가될 타입:

- `profiles.role` 필드 추가
- `events` 테이블 (Row/Insert/Update)
- `event_participants` 테이블 (Row/Insert/Update)

### Step 6: 프론트엔드 임시 타입 교체

**수정 파일 목록**:

## 검증 방법

1. **마이그레이션 적용 확인**: `mcp__supabase__list_tables`로 3개 테이블 존재 확인
2. **RLS 정책 확인**: `mcp__supabase__get_advisors({ type: 'security' })`로 RLS 경고 없는지 확인
3. **인덱스 확인**: `mcp__supabase__execute_sql`로 `pg_indexes` 쿼리
4. **타입 컴파일**: `npm run typecheck` 통과
5. **전체 품질 검사**: `npm run check-all` 통과

## 주의사항

- `handle_updated_at()` 함수는 profiles 마이그레이션에서 이미 생성됨 → events 테이블 트리거에서 `CREATE OR REPLACE` 없이 트리거만 추가
- `event.date` → `event.event_date` 변경 시 `formatEventDate()`, `isoToDatetimeLocal()` 함수 인수 수정도 필요 (함수 자체 로직 변경 없음, 호출 코드만 수정)
- Storage 버킷 생성은 MCP 또는 Supabase 대시보드 사용 (SQL로는 불가)
- `database.types.ts`는 마이그레이션 적용 완료 후 재생성
